# AHK Manager — Архитектура магазина скриптов

## Вводная

Магазин скриптов — это встроенная в AHK Manager система для поиска, установки и обновления AutoHotkey-скриптов от сообщества. Главный принцип: **никакого собственного сервера**. Вся инфраструктура строится на GitHub: репозиторий = база данных, `raw.githubusercontent.com` = CDN для скачивания, Pull Request = процесс публикации, GitHub Actions = CI/CD.

Такой подход не выдуман нами — это проверенный паттерн, которым пользуются Obsidian (500+ плагинов), Scoop (1300+ пакетов), Flow Launcher, Espanso и другие. Все они сходятся к одной схеме: один JSON-файл как каталог + GitHub для хостинга файлов.

### Почему именно так?

- **Бесплатно** — GitHub бесплатен для open source, нет серверов для оплаты
- **Надёжно** — GitHub CDN с кешированием через Fastly, аптайм 99.9%+
- **Просто** — не нужно писать и поддерживать API-сервер, базу данных, аутентификацию
- **Прозрачно** — весь каталог виден как обычный GitHub-репозиторий, любой может проверить
- **Масштабируемо** — Scoop обслуживает тысячи пакетов на той же архитектуре

### Объяснение ключевых решений

**Почему `catalog.json` а не API?**
Один JSON-файл на 1000 скриптов весит ~60KB в сжатом виде. Один HTTP-запрос — и у нас весь каталог. Не нужно пагинировать, не нужен API-сервер. `raw.githubusercontent.com` не имеет rate limits для публичных репозиториев и поддерживает ETag-кеширование (если файл не изменился — ответ `304 Not Modified` мгновенный и бесплатный).

**Почему директория на каждый скрипт, а не плоский список?**
Многие AHK-скрипты используют `#Include` для подключения библиотек. Директория-на-скрипт позволяет хранить рядом с главным `.ahk` файлом его зависимости (`lib/utils.ahk`), ресурсы (иконки, DLL) и документацию (README, скриншоты). Это как npm-пакет — всё в одной папке.

**Почему публикация через Pull Request?**
Это проверенный механизм модерации. Автор форкает репозиторий, добавляет свой скрипт, открывает PR. Автоматически запускается CI (проверка синтаксиса, security-сканирование). Мейнтейнер проверяет вручную. После мержа GitHub Action автоматически пересобирает `catalog.json`. Homebrew, Scoop, Winget — все работают именно так.

**Почему SQLite для tracking установок?**
У нас уже есть SQLite в приложении (`ahkmanager.db`). Добавляем одну таблицу `marketplace_scripts` для отслеживания: что установлено, какая версия, откуда скачано, хеш для проверки целостности.

## 2. Формат манифеста (`manifest.json`)

Каждый скрипт в репозитории магазина сопровождается файлом `manifest.json` — это "паспорт" скрипта. Формат вдохновлён npm `package.json`, Scoop manifests, VS Code extension manifest и Tampermonkey userscript headers.

### Полный пример

```json
{
  "name": "window-snapper",
  "displayName": "Window Snapper",
  "version": "1.3.0",
  "description": "Привязка окон к четвертям и третям экрана горячими клавишами",
  "author": {
    "name": "Alex Kozlov",
    "github": "alexkoz"
  },
  "ahk": {
    "version": "v2",
    "minVersion": "2.0.10",
    "entryFile": "window-snapper.ahk",
    "hasGui": false,
    "runsResident": true,
    "hotkeys": [
      { "keys": "#!Left", "action": "Привязать окно к левой половине" },
      { "keys": "#!Right", "action": "Привязать окно к правой половине" },
      { "keys": "#!Up", "action": "Привязать окно к верхней половине" },
      { "keys": "#!Down", "action": "Привязать окно к нижней половине" }
    ]
  },
  "permissions": {
    "requiresAdmin": false,
    "network": false,
    "clipboard": false,
    "registry": false,
    "shellExecute": false,
    "keyboardHook": true,
    "mouseHook": false
  },
  "categories": ["productivity"],
  "tags": ["window-management", "tiling", "snap", "hotkeys"],
  "license": "MIT",
  "readme": "README.md",
  "icon": "icon.png",
  "screenshots": [
    { "path": "screenshots/demo.png", "caption": "Привязка к четвертям экрана" }
  ],
  "repository": {
    "type": "git",
    "url": "https://github.com/alexkoz/window-snapper"
  },
  "files": ["window-snapper.ahk", "README.md", "LICENSE"],
  "changelog": "Добавлена поддержка нескольких мониторов"
}
```

### Разбор каждого поля

#### Идентификация (обязательные поля)

| Поле | Тип | Описание |
|------|-----|----------|
| `name` | string | Уникальный slug скрипта. Только `[a-z0-9-]`, максимум 50 символов. Это ID скрипта в каталоге — как имя npm-пакета. Пример: `"clipboard-manager"` |
| `displayName` | string | Человекочитаемое название для отображения в UI. Пример: `"Clipboard Manager Pro"` |
| `version` | string | Версия в формате semver (`1.2.3`). Bumрится автором при каждом обновлении. CI проверяет что версия увеличилась |
| `description` | string | Краткое описание в одну строку, до 200 символов. Показывается в карточке скрипта в каталоге |

#### Авторство

| Поле | Тип | Описание |
|------|-----|----------|
| `author.name` | string | **Обязательно.** Имя автора |
| `author.github` | string? | GitHub username — используется для ссылки на профиль и верификации |
| `author.email` | string? | Контактный email (опционально) |
| `author.url` | string? | Личный сайт автора (опционально) |
| `contributors` | array? | Список соавторов: `[{ "name": "...", "github": "..." }]` |

#### AHK-специфичные поля (критически важно)

Сгруппированы в объект `ahk` — это всё, что касается AutoHotkey как рантайма:

| Поле | Тип | Описание |
|------|-----|----------|
| `ahk.version` | string | **Обязательно.** `"v1"`, `"v2"` или `"v1+v2"`. AHK v1 и v2 абсолютно несовместимы — скрипт v1 не запустится на v2 и наоборот. Это главный фильтр в магазине |
| `ahk.minVersion` | string? | Минимальная версия AHK, например `"2.0.10"`. Позволяет предупредить пользователя если его AHK слишком старый |
| `ahk.entryFile` | string | **Обязательно.** Главный `.ahk` файл, который запускается. Относительный путь от корня директории скрипта |
| `ahk.hasGui` | boolean | Имеет ли скрипт графический интерфейс (окна, диалоги). По умолчанию `false`. Влияет на UX: для GUI-скриптов показываем скриншоты, кнопку "Показать UI" |
| `ahk.runsResident` | boolean | Остаётся ли скрипт в трее (резидентный) или выполняется и завершается. По умолчанию `false`. Резидентные скрипты требуют другого подхода к обновлению (нужно остановить → обновить → перезапустить) |
| `ahk.hotkeys` | array? | Список горячих клавиш скрипта. Каждый элемент: `{ "keys": "#!Left", "action": "Описание действия" }`. Позволяет показать пользователю хоткеи ДО установки и проверить конфликты с его существующими скриптами |
| `ahk.directives` | array? | Директивы AHK: `["#Requires AutoHotkey v2.0+"]`. Парсятся из скрипта для информации |

**Почему `ahk.version` — самое важное поле?**
AHK-сообщество разделено между v1 и v2. Синтаксис кардинально отличается. Если пользователь использует AHK v2, показывать ему скрипты для v1 бессмысленно — они не запустятся. Это первый и главный фильтр, без него магазин бесполезен.

**Почему `hotkeys` в манифесте?**
Уникальная особенность AHK-скриптов. Ни один другой пакетный менеджер этого не делает, потому что у npm/pip нет горячих клавиш. Но для AHK это критично: пользователь может установить два скрипта с одинаковым `Win+Space` — и получить конфликт. Декларация хоткеев в манифесте позволяет:
1. Показать хоткеи на странице скрипта до установки
2. Предупредить о конфликтах с уже установленными скриптами
3. Построить глобальную карту хоткеев в приложении

#### Permissions (разрешения)

AHK-скрипты имеют **полный доступ** к системе — они могут симулировать ввод, читать/писать файлы, лезть в реестр, качать файлы из интернета, запускать любые программы. Песочницы нет и быть не может (это противоречит самой сути AHK). Поэтому вместо песочницы мы используем **декларацию разрешений** — автор явно указывает что делает его скрипт, а мы показываем это пользователю перед установкой.

Вдохновлено Tampermonkey (`@grant`) и Chrome Extension permissions.

| Разрешение | Что означает | Пример легитимного использования |
|------------|-------------|--------------------------------|
| `requiresAdmin` | Запуск от администратора | Управление системными службами, доступ к защищённым путям |
| `network` | HTTP-запросы, скачивание файлов | Проверка обновлений, API-клиенты, загрузчики |
| `clipboard` | Чтение/запись буфера обмена | Clipboard managers, text expanders |
| `registry` | Чтение/запись реестра Windows | Настройка системных параметров, ассоциации файлов |
| `shellExecute` | Запуск внешних программ (`Run`, `RunWait`) | Открытие файлов, вызов утилит |
| `keyboardHook` | Перехват нажатий клавиатуры | Горячие клавиши (большинство AHK-скриптов) |
| `mouseHook` | Перехват событий мыши | Жесты мышью, auto-clickers |
| `fileSystem` | Чтение/запись файлов вне своей директории | Бэкапы, обработка файлов, логирование |

**Как это работает в UI:**

Перед установкой пользователь видит цветовую индикацию:
- **Зелёный (Safe):** только `keyboardHook` — обычный hotkey-скрипт, безопасно
- **Жёлтый (Caution):** `clipboard`, `shellExecute`, `fileSystem` — скрипт делает больше чем просто хоткеи, стоит посмотреть исходный код
- **Красный (Dangerous):** `network` + `shellExecute` (может скачать и запустить что угодно), `registry` (может изменить систему), `requiresAdmin` (полный доступ)

#### Категории (фиксированный список)

Авторы выбирают 1-2 категории из фиксированного списка. Категории используются для навигации в магазине (горизонтальные табы сверху).

| Категория | Описание | Примеры скриптов |
|-----------|----------|-----------------|
| `productivity` | Управление окнами, буфер обмена, текстовые инструменты | Window Snapper, Clipboard History, Text Expander |
| `automation` | Автоматизация файловых операций, batch-задачи | File Renamer, Backup Tool, Folder Watcher |
| `gaming` | Игровые макросы, overlays, ремапинг для игр | Auto Clicker, Game Overlay, Anti-AFK |
| `accessibility` | Вспомогательные технологии | Screen Magnifier, High Contrast Toggle, Sticky Keys |
| `dev-tools` | Инструменты для разработчиков | Color Picker, Regex Tester, Git Shortcuter |
| `media` | Аудио/видео управление, скриншоты | Volume Mixer, Screenshot Tool, Media Controls |
| `system` | Системные утилиты, трей-инструменты | CPU Monitor, Tray Clock, Power Manager |
| `input` | Ремапинг клавиш/мыши (не для игр) | CapsLock→Ctrl, Custom Keyboard Layout |
| `libraries` | Переиспользуемые AHK-библиотеки (не самостоятельные скрипты) | JSON Parser, HTTP Library, GUI Framework |

**Теги (`tags`)** — свободные ключевые слова для детального поиска. В отличие от категорий, теги не фиксированы. Максимум 10 тегов. Показываются как чипы на карточке скрипта.

#### Прочие поля

| Поле | Описание |
|------|----------|
| `license` | SPDX-идентификатор лицензии: `"MIT"`, `"GPL-3.0-only"`, `"UNLICENSED"`. **Обязательно** — без лицензии скрипт не принимается |
| `readme` | Путь к README.md (рендерится как описание на странице скрипта) |
| `icon` | Путь к иконке (256x256 PNG, показывается в каталоге) |
| `screenshots` | Массив скриншотов для GUI-скриптов: `[{ "path": "...", "caption": "..." }]` |
| `repository` | Ссылка на исходный репозиторий автора |
| `files` | Список файлов для включения в пакет. Glob-паттерны: `["*.ahk", "lib/**"]` |
| `changelog` | Описание изменений в текущей версии (показывается при обновлении) |

## 3. Каталог (`catalog.json`)

Это единый индексный файл, содержащий **облегчённые метаданные всех скриптов** в магазине. Генерируется автоматически GitHub Actions при каждом мерже в main.

### Зачем нужен отдельный каталог, если есть manifest.json?

`manifest.json` — полный паспорт одного скрипта (может быть большим, с README, скриншотами). `catalog.json` — это лёгкий индекс: только то что нужно для отображения в списке и поиска. Приложение скачивает один `catalog.json` вместо тысячи отдельных `manifest.json`.

Аналогия: `catalog.json` — это таблица содержания книги, а `manifest.json` — полная глава.

### Структура

```json
{
  "version": 2,
  "updatedAt": "2026-04-06T12:00:00Z",

  "scripts": [
    {
      "id": "window-snapper",
      "name": "Window Snapper",
      "description": "Привязка окон к четвертям экрана горячими клавишами",
      "author": "alexkoz",
      "version": "1.3.0",
      "ahkVersion": "v2",
      "category": "productivity",
      "tags": ["window-management", "tiling"],
      "hasGui": false,
      "runsResident": true,
      "hotkeys": ["#!Left", "#!Right", "#!Up", "#!Down"],
      "permissions": ["keyboardHook"],
      "license": "MIT",
      "size": 4096,
      "sha": "a1b2c3d4",

    }
  ],

  "categories": [
    { "id": "productivity", "name": "Productivity", "nameRu": "Продуктивность", "count": 45 },
    { "id": "gaming", "name": "Gaming", "nameRu": "Игры", "count": 23 },
    { "id": "system", "name": "System", "nameRu": "Система", "count": 18 }

    {
      "id": "essentials",

    }
  ]

```

### Объяснение полей каталога

**`scripts[]`** — облегчённая версия manifest. Содержит только то что нужно для:
- Отображения карточки (name, description, author, icon)
- Фильтрации (ahkVersion, category, tags, permissions)
- Сортировки (downloads, rating, createdAt, updatedAt)
- Проверки обновлений (version, sha)

**`categories[]`** — список категорий с количеством скриптов и русскими названиями. Генерируется автоматически из manifest-файлов.

**`collections[]`** — кураторские подборки. Создаются вручную мейнтейнерами. Отображаются на главной странице магазина как "Featured" и "Рекомендуемые".

### Размер и производительность

| Количество скриптов | Размер raw | Размер gzipped | Время скачивания (10 Мбит/с) |
|---|---|---|---|
| 100 | ~40 КБ | ~6 КБ | мгновенно |
| 500 | ~200 КБ | ~30 КБ | мгновенно |
| 1000 | ~400 КБ | ~60 КБ | <1 сек |
| 5000 | ~2 МБ | ~300 КБ | 2-3 сек |

Даже на 5000 скриптов — один HTTP-запрос и пара секунд. Этого хватит на годы роста.

### Стратегия обновления каталога

```
Запуск приложения
    │
    ├─ GET catalog.json с заголовком If-None-Match: "<сохранённый_etag>"
    │
    ├─ Ответ 304 Not Modified?
    │   └─ Да → используем локальный кеш, запрос бесплатный (не считается в rate limit)
    │   └─ Нет (200) → скачиваем новый каталог, сохраняем ETag
    │
    └─ Повторная проверка каждые 6 часов в фоне (если приложение открыто)
```

**Что такое ETag?** Это "отпечаток" файла. GitHub возвращает его в заголовке ответа. При следующем запросе мы отправляем этот отпечаток — если файл не изменился, GitHub отвечает `304 Not Modified` без тела ответа. Экономит трафик и не расходует лимит API-запросов.

**Офлайн-режим:** Каталог кешируется в localStorage (фронтенд) и в SQLite (бэкенд). Если сети нет — пользователь видит последний загруженный каталог. Установка и скачивание недоступны, но можно просматривать каталог и управлять уже установленными скриптами.

## 4. Архитектура фронтенда

### Zustand Store

Магазин получает **отдельный** Zustand store, не расширяя существующий `useTreeStore`. Причина: у них нет общего состояния. `useTreeStore` управляет деревом локальных скриптов (папки, vim-режим, теги, drag-and-drop). Marketplace store управляет каталогом, поиском, установкой — это другой домен.

```typescript
// src/store/useMarketplaceStore.ts

interface MarketplaceStore {
  // ── Каталог ──────────────────────────────────────────
  items: CatalogScript[];        // массив скриптов из catalog.json
  lastFetchedAt: number | null;  // когда последний раз обновляли каталог
  isFetching: boolean;           // идёт ли загрузка

  // ── Фильтры и поиск ─────────────────────────────────
  searchQuery: string;           // текст в поисковой строке
  category: string | null;       // выбранная категория (или null = все)
  ahkVersionFilter: "v1" | "v2" | null;  // фильтр по версии AHK
  sortBy: "popular" | "recent" | "name" | "rating";  // сортировка

  // ── Навигация внутри магазина ────────────────────────
  storeView: "browse" | "detail" | "installed";  // текущий "экран"
  detailId: string | null;       // ID скрипта для детальной страницы

  // ── Установленные скрипты ────────────────────────────
  installed: Record<string, InstalledMeta>;  // что установлено, ключ = scriptId
  operations: InstallOp[];       // очередь операций (install/update/uninstall)

  // ── Сеть ─────────────────────────────────────────────
  isOnline: boolean;             // есть ли интернет
}
```

**Что сохраняется между перезапусками (localStorage):**
- `installed` — список установленных скриптов (должен пережить перезапуск)
- `items` — кешированный каталог (для офлайн-режима)
- `lastFetchedAt` — для решения "стоит ли обновлять каталог"
- `sortBy` — пользовательская настройка

**Что НЕ сохраняется (сбрасывается при перезапуске):**
- `searchQuery` — каждый раз с чистого листа
- `storeView`, `detailId` — навигация начинается с browse
- `operations` — текущие операции не переживают перезапуск
- `isFetching` — техническое состояние

### Поиск: MiniSearch

Из четырёх исследованных библиотек (Fuse.js, FlexSearch, Lunr.js, MiniSearch) выбран **MiniSearch** как лучший баланс:

| Критерий | MiniSearch | Альтернативы |
|----------|-----------|-------------|
| Размер | **4 КБ** gzipped | Fuse: 5KB, FlexSearch: 6-22KB, Lunr: 8KB |
| Fuzzy-поиск | Хороший | Fuse: лучший, FlexSearch: средний |
| Auto-suggest | **Встроенный** | Нет ни у кого другого |
| Мутабельный индекс | **Да** (можно добавлять/удалять) | Lunr: нет (immutable) |
| Подсветка совпадений | **Да** (возвращает match data) | FlexSearch: нет |
| API | **Самый чистый** | FlexSearch: сложный API |

```typescript
// Настройка индекса
const index = new MiniSearch({
  fields: ['name', 'description', 'tags_joined', 'author'],
  storeFields: ['name', 'description', 'author', 'category', 'downloads'],
  searchOptions: {
    // Вес полей: название важнее всего, потом теги, потом описание
    boost: { name: 3, tags_joined: 2, description: 1.5 },
    fuzzy: 0.2,     // толерантность к опечаткам (0 = точный, 1 = любое совпадение)
    prefix: true,   // "clip" найдёт "clipboard"
  },
});

// Поиск
const results = index.search('clipbord manag'); // найдёт "Clipboard Manager" несмотря на опечатку

// Подсказки "Вы имели в виду?"
const suggestions = index.autoSuggest('clip'); 
// → [{ suggestion: 'clipboard manager', score: 8.2 }]
```

**Debounce:** Используем `useDeferredValue` из React 19 — нативный способ отложить ре-рендер при быстром вводе. Не нужен отдельный таймер или `lodash.debounce`.

### Навигация

Магазин — новый таб в сайдбаре, на одном уровне с Hub, All и Settings:

```
Sidebar:
  [Hub]          ← избранные скрипты
  [All]          ← все локальные скрипты
  [Untagged]     ← без тегов
  --- теги пользователя ---
  [PoE]
  [System]
  --- разделитель ---
  [Store] (3)    ← магазин, badge = количество доступных обновлений
  [Settings]     ← настройки
```

Внутри магазина — своя навигация через `storeView`:
- **browse** — главная: featured + trending + категории + поиск
- **detail** — страница конкретного скрипта (README, скриншоты, хоткеи, permissions)
- **installed** — список установленных скриптов с проверкой обновлений

### Новая зависимость: TanStack Query (React Query)

**13 КБ** gzipped. Зачем:
- Автоматическое кеширование и дедупликация запросов (не скачивает каталог дважды)
- `stale-while-revalidate` — показывает старые данные мгновенно, обновляет в фоне
- `useInfiniteQuery` — для бесконечного скролла (если понадобится пагинация)
- `useMutation` — для install/uninstall с оптимистичными обновлениями и откатом при ошибке
- Автоматический retry при сетевых ошибках

Без React Query пришлось бы вручную писать кеширование, дедупликацию, retry логику и оптимистичные обновления — это ~500 строк бойлерплейта.

## 6. Rust Backend

### Новые Tauri-команды

```rust
// ── Каталог ─────────────────────────────────────────
marketplace_get_catalog()
  // Скачивает catalog.json (или возвращает из кеша)
  // Использует ETag для условных запросов
  // → Catalog { scripts: Vec<CatalogScript>, categories, collections }

marketplace_search(query, category, sort, page)
  // Фильтрация и сортировка каталога на Rust-стороне
  // (или можно делать на фронте через MiniSearch)
  // → CatalogPage { items, total, hasMore }

// ── Установка ───────────────────────────────────────
marketplace_install_script(id)
  // 1. Находит скрипт в кешированном каталоге
  // 2. Скачивает файлы с raw.githubusercontent.com
  // 3. Стримит прогресс через Tauri events
  // 4. Проверяет SHA-256 хеш
  // 5. Записывает во временный файл
  // 6. Атомарный rename в целевую директорию
  // 7. Создаёт запись в marketplace_scripts таблице
  // 8. Эмитит marketplace-install-complete
  // 9. Триггерит обновление дерева скриптов
  // → Result<String, String> (install path или ошибка)

marketplace_uninstall_script(id, deleteFile: bool)
  // Удаляет из БД. Если deleteFile — удаляет файл с диска.
  // → Result<(), String>

marketplace_update_script(id)
  // 1. Останавливает скрипт если запущен
  // 2. Сохраняет бэкап (.ahk.bak)
  // 3. Скачивает новую версию
  // 4. Атомарная замена файла
  // 5. Перезапускает если был запущен
  // → Result<(), String>

marketplace_check_updates()
  // Сравнивает installed versions с catalog.json
  // → Vec<UpdateInfo { id, name, currentVersion, latestVersion, changelog }>

marketplace_rollback_script(id)
  // Восстанавливает .ahk.bak → .ahk
  // → Result<(), String>

// ── Детали ──────────────────────────────────────────
marketplace_get_script_detail(id)
  // Скачивает полный manifest.json + README.md
  // → ScriptDetail { manifest, readme: String, sourceCode: String }

marketplace_get_script_source(id)
  // Скачивает исходный код .ahk файла для превью
  // → String
```

### Расширение базы данных

Добавляем одну таблицу в существующую SQLite БД (`ahkmanager.db`):

```sql
CREATE TABLE IF NOT EXISTS marketplace_scripts (
    -- Идентификация
    marketplace_id    TEXT PRIMARY KEY,    -- ID скрипта в каталоге ("window-snapper")
    script_id         TEXT,               -- FK на scripts(id) — связь с локальным скриптом
    
    -- Расположение
    install_path      TEXT NOT NULL,       -- куда установлен на диске
    source_url        TEXT NOT NULL,       -- откуда скачан (raw.githubusercontent URL)
    
    -- Версионирование
    version           TEXT NOT NULL,       -- установленная версия ("1.3.0")
    prev_version      TEXT,               -- предыдущая версия (для отката)
    content_hash      TEXT NOT NULL,       -- SHA-256 установленного файла
    prev_content_hash TEXT,               -- SHA-256 предыдущей версии (для отката)
    
    -- Timestamps
    installed_at      TEXT NOT NULL,       -- когда установлен (ISO 8601)
    updated_at        TEXT,               -- когда последний раз обновлён
    last_check_at     TEXT,               -- когда проверяли обновления
    
    -- Настройки
    last_etag         TEXT,               -- ETag для conditional requests
    auto_update       INTEGER DEFAULT 0   -- автообновление (0/1)
);
```

**Связь с существующей таблицей `scripts`:**
При установке скрипт из магазина попадает в сканируемую директорию → `get_scripts` находит его → создаёт запись в таблице `scripts`. `marketplace_scripts.script_id` ссылается на `scripts.id`. Это позволяет marketplace-скрипту работать как обычный локальный скрипт (запуск, остановка, теги) плюс иметь дополнительные метаданные (версия, источник, обновления).

### Директория установки

```
<первый_scan_path>/marketplace/
  window-snapper/
    window-snapper.ahk
  clipboard-manager/
    ClipboardManager.ahk
    lib/
      utils.ahk
```

**Почему отдельная папка `marketplace/`:**
- Чёткое разделение пользовательских и магазинных скриптов
- Упрощает обновление и откат (мы "владеем" этой папкой)
- Пользователь не перепутает и не отредактирует магазинный скрипт случайно
- При uninstall — просто удаляем директорию скрипта

Папка `marketplace/` добавляется как implicit scan path (не отображается в настройках, но сканируется автоматически).

### Процесс скачивания (подробно)

```
1. Frontend вызывает invoke("marketplace_install_script", { id: "window-snapper" })

2. Rust находит скрипт в кешированном каталоге по ID

3. Формирует URL:
   https://raw.githubusercontent.com/ahk-community/script-store/main/scripts/window-snapper/window-snapper.ahk
   
   raw.githubusercontent.com — это CDN GitHub:
   - Без rate limits для публичных репо
   - Без авторизации
   - Кешируется на уровне CDN (Fastly)
   - Поддерживает ETag для условных запросов

4. reqwest скачивает файл стримом (chunks):
   - Каждые 50мс эмитит Tauri event "marketplace-download-progress"
     { scriptId, downloaded, total, phase: "downloading" }
   - Frontend показывает прогресс в кнопке Install

5. После скачивания — проверка SHA-256:
   - Хеш из catalog.json сравнивается с хешем скачанного файла
   - Если не совпадает → ошибка HashMismatch (файл повреждён или подменён)
   - Эмитит phase: "verifying"

6. Запись во временный файл:
   .ahkm_tmp_<uuid> в той же директории (тот же filesystem для атомарного rename)

7. Атомарный rename:
   .ahkm_tmp_<uuid> → window-snapper.ahk
   
   Если файл уже существует (обновление) — сначала rename в .ahk.bak (бэкап)
   
   Rename на одном filesystem — атомарная операция. Либо старый файл,
   либо новый. Никогда не будет "наполовину скачанного" файла.

8. Запись в SQLite:
   INSERT INTO marketplace_scripts (marketplace_id, install_path, version, content_hash, ...)

9. Emit event:
   "marketplace-install-complete" { marketplace_id, path }
   Frontend обновляет состояние кнопки Install → Installed

10. Trigger scan:
    Вызывается get_scripts(forceScan: true) чтобы новый скрипт
    появился в основном дереве скриптов
```

### Проверка обновлений

```
1. Запуск приложения → 5 секунд задержка (чтобы UI загрузился первым)

2. GET catalog.json (с ETag — conditional request)

3. Для каждого установленного скрипта:
   if catalog_version != installed_version → добавить в список обновлений

4. Emit "marketplace-updates-available" с массивом UpdateInfo

5. Frontend:
   - Показывает badge на иконке Store в сайдбаре: "Store (3)"
   - В разделе "Installed" — кнопки Update рядом с устаревшими скриптами
   - Кнопка "Обновить всё" наверху
```

## 7. Модель безопасности

### Почему безопасность критична

AHK-скрипты выполняются с **полными правами пользователя**. Они могут:
- Записывать каждое нажатие клавиши (кейлоггер)
- Скачивать и запускать произвольные программы
- Читать/писать любые файлы
- Модифицировать реестр Windows
- Отправлять данные по сети

**Песочница невозможна** — это противоречит самой природе AHK (инструмент для автоматизации ввода и управления системой). Поэтому модель безопасности строится на: **прозрачности** (показать что делает скрипт), **модерации** (проверить перед публикацией), **реагировании** (быстро убрать вредоносный скрипт).

### Уровни доверия

| Уровень | Кто | Как проверяется | Задержка публикации |
|---------|-----|-----------------|-------------------|
| **Unverified** | Новый автор, первый скрипт | Полная ручная проверка + автотесты | 72 часа |
| **Community** | 1+ скрипт прошёл review без проблем | Автоматические проверки + 1 reviewer | 24 часа |
| **Trusted** | 3+ скриптов без инцидентов, аккаунт >6 мес | Auto-merge если CI зелёный | Мгновенно |
| **Curated** | Core team, проверенные партнёры | Pre-approved | Мгновенно |

**Почему 72 часа для новых авторов?** Это время для ручного review. Если за 72 часа мейнтейнер не посмотрел — автоматического мержа НЕ будет. Скрипт от нового автора всегда требует человеческих глаз.

### Автоматическое сканирование (CI)

GitHub Actions запускается на каждый PR, проверяет AHK-файлы на подозрительные паттерны:

**CRITICAL (блокирует PR автоматически):**
```
Download + Run         — скачивание + запуск = потенциальный dropper
URLDownloadToFile      — скачивание файлов из интернета
RunWait.*powershell.*-enc  — закодированные PowerShell-команды (обфускация)
RegWrite.*\Run         — запись в автозапуск реестра (персистентность)
FileInstall            — встроенный бинарник (потенциальный payload)
```

**HIGH (помечает для ручной проверки, не блокирует):**
```
Run/RunWait с переменными    — динамический запуск программ
RegWrite/RegDelete            — любые изменения реестра
DllCall с числовыми адресами  — прямые системные вызовы (обфускация)
ClipboardAll                  — мониторинг всего буфера обмена
```

Эти паттерны легитимны во многих скриптах! Поэтому они не блокируют PR, а помечают для ручной проверки. Reviewer видит: "этот скрипт использует Run — это ожидаемо для его функциональности?"

**Также проверяется:**
- Валидность manifest.json (JSON Schema)
- Синтаксис AHK (`AutoHotkey.exe /validate`)
- Размер файлов (<500 КБ на скрипт)
- Уникальность ID скрипта
- Что версия увеличилась (для обновлений)

### Отображение разрешений

Перед установкой пользователь видит "светофор":

```
🟢 Safe (безопасно):
   Только keyboardHook — обычный скрипт с горячими клавишами.
   Не лезет в сеть, не запускает программы, не трогает файлы.
   
🟡 Caution (осторожно):
   clipboard, shellExecute, fileSystem — скрипт делает больше
   чем просто хоткеи. Рекомендуется посмотреть исходный код.
   
🔴 Dangerous (опасно):
   network + shellExecute — может скачать и запустить что угодно.
   registry — может изменить систему.
   requiresAdmin — полный доступ к системе.
   
   Показывается предупреждающий диалог.
```

### Blocklist (экстренное удаление)

Если обнаружен вредоносный скрипт:

```json
{
  "blocked": [
    {
      "scriptId": "malicious-script",
      "versions": ["1.2.0", "1.2.1"],
      "reason": "Кража данных через WinHTTP",
      "severity": "critical",
      "action": "kill_and_warn"
    }
  ]
}
```

**Как работает:**
1. Мейнтейнер добавляет скрипт в blocklist.json (<1 часа после обнаружения)
2. При запуске приложения — проверка blocklist
3. Если установленный скрипт в blocklist → автоматическая остановка процесса
4. Показ предупреждения: "Скрипт X был удалён из магазина по причине безопасности"
5. Кнопки: "Остановить и удалить" / "Подробнее" / "Пропустить"

### Исходный код как главный сигнал доверия

Уникальная возможность магазина скриптов (vs app store / npm): **пользователь может прочитать весь исходный код прямо в приложении** на вкладке "Исходный код" страницы скрипта. AHK-скрипты — это текстовые файлы на ~100-500 строк. Любой опытный пользователь может за минуту проверить что скрипт не делает ничего подозрительного.

В приложении уже есть подсветка AHK-синтаксиса (реализована для просмотра локальных скриптов) — переиспользуем её.

## 8. CI/CD на GitHub Actions

### Валидация Pull Request

```yaml
# .github/workflows/validate-pr.yml
name: Validate Script
on:
  pull_request:
    paths: ['scripts/**']

jobs:
  validate:
    runs-on: windows-latest    # Windows нужен для AutoHotkey.exe /validate
    steps:
      - uses: actions/checkout@v4
      
      - name: Проверка manifest.json
        # JSON Schema валидация: все обязательные поля,
        # правильные типы, name уникален, version semver
        
      - name: Проверка синтаксиса AHK
        # Скачивает portable AHK v2
        # Запускает AutoHotkey64.exe /validate на каждом .ahk файле
        # /validate проверяет синтаксис без запуска скрипта
        
      - name: Security scan
        # Grep по подозрительным паттернам
        # CRITICAL → fail PR
        # HIGH → добавить label "security-review-needed"
        
      - name: Проверка ограничений
        # Размер файлов < 500KB
        # Все файлы из manifest.files существуют
        # ID скрипта не дублирует существующий
        
      - name: Авто-маркировка
        # Добавляет labels: "category:productivity", "ahk-v2"
```

### Пересборка каталога

```yaml
# .github/workflows/build-catalog.yml
name: Build Catalog
on:
  push:
    branches: [main]
    paths: ['scripts/**']

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Генерация catalog.json
        # Python-скрипт:
        # 1. Обходит все scripts/*/manifest.json
        # 2. Извлекает облегчённые метаданные
        # 3. Считает хеши файлов
        # 4. Генерирует categories[] с подсчётом
        # 5. Включает collections[] из curated-collections.json
        # 6. Записывает catalog.json
        
      - name: Коммит catalog.json
        # git add catalog.json
        # git commit -m "chore: rebuild catalog"
        # git push
        #
        # После push — raw.githubusercontent.com обновляется за ~30 секунд
        # Следующий запрос от приложения получит 200 вместо 304
```

## 9. Как авторы публикуют скрипты

### Phase 1: Через GitHub (MVP)

Простейший вариант — как в Homebrew, Scoop, Winget:

```
1. Автор форкает https://github.com/ahk-community/script-store

2. Создаёт директорию для своего скрипта:
   scripts/
     my-awesome-script/
       manifest.json      ← заполняет по шаблону
       my-awesome-script.ahk
       README.md           ← описание (опционально)

3. Открывает Pull Request

4. Автоматически запускается CI:
   ✓ manifest.json валиден
   ✓ AHK синтаксис проверен
   ⚠ Security scan: используется Run (помечено для review)

5. Мейнтейнер проверяет:
   - Скрипт делает то что описано?
   - Нет ли подозрительного кода?
   - Описание и метаданные корректны?

6. Merge → catalog.json пересобирается автоматически
   → скрипт появляется в магазине через ~1 минуту
```

**Инструкция для авторов** — простой README в корне репозитория с пошаговым гайдом и шаблоном manifest.json.

### Phase 2: Из приложения

Когда магазин наберёт аудиторию:

```
1. Пользователь нажимает "Опубликовать" в меню скрипта

2. Приложение запрашивает авторизацию GitHub:
   - GitHub OAuth Device Flow
   - Показывает код: "Введите ABCD-1234 на github.com/login/device"
   - Пользователь авторизует в браузере
   - Токен сохраняется в OS Keyring (Windows Credential Manager)

3. Форма публикации:
   - Название, описание, категория, теги
   - Версия AHK (v1/v2)
   - Горячие клавиши (парсятся автоматически)
   - Лицензия (dropdown)

4. Приложение через GitHub API:
   - Форкает repo (если ещё нет форка)     — 1 API-вызов
   - Создаёт branch                         — 1 API-вызов
   - Загружает файлы                         — 1-2 API-вызова
   - Открывает PR с заполненным описанием   — 1 API-вызов
   
   Итого: 4-5 API-вызовов. В рамках лимита 5000/час.

5. Пользователь видит ссылку на PR
   Дальше — стандартный review + merge процесс
```

## 10. Технические зависимости

### Frontend — новые пакеты

| Пакет | Размер (gzipped) | Зачем нужен | Альтернативы и почему не они |
|-------|-----------------|-------------|------------------------------|
| `minisearch` | 4 КБ | Полнотекстовый поиск в каталоге | Fuse.js (5KB, нет auto-suggest), FlexSearch (22KB, сложный API), Lunr (8KB, immutable index) |
| `@tanstack/react-query` | 13 КБ | Кеширование запросов, optimistic updates, retry | SWR (нет mutations), ручной fetch (500 строк бойлерплейта) |

**Итого:** +17 КБ к бандлу. Для Tauri-приложения (не веб) это несущественно.

### Rust — уже в проекте

Все нужные crates уже есть в `Cargo.toml`:
- `reqwest` (с features `stream`, `native-tls`) — HTTP-клиент для скачивания
- `sha2` — SHA-256 хеширование для проверки целостности
- `serde` + `serde_json` — сериализация/десериализация JSON
- `rusqlite` (с feature `bundled`) — SQLite для tracking
- `uuid` — генерация ID для скриптов
- `tokio` — async runtime для фоновых задач
- `futures-util` — для стриминга скачивания (`.bytes_stream()`)

### Новый Rust crate (Phase 2)

| Crate | Зачем |
|-------|-------|
| `keyring` | Хранение GitHub OAuth токена в Windows Credential Manager. Безопаснее чем localStorage или plaintext файл |

### Главный вывод

**Все zero-server проекты сходятся к одному и тому же паттерну:**

1. Один JSON-файл как индекс всех пакетов в GitHub-репозитории
2. `raw.githubusercontent.com` или GitHub Releases для скачивания файлов
3. PR-based публикация с CI-валидацией
4. Локальная SQLite/JSON для tracking установленного

Obsidian доказал что этот подход работает на 500+ плагинов. Scoop — на 1300+ пакетов. Нет причин изобретать что-то другое.
