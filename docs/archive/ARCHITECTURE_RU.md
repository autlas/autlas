# Архитектура — AHK Manager

## Общая картина

Десктопное приложение для управления AutoHotkey-скриптами.

**Стек:** Tauri 2 (Rust бекенд) + React 19 + TypeScript + Tailwind CSS.

**Как всё связано:** пользователь нажимает кнопку в интерфейсе — React-компонент вызывает Rust-команду через IPC — Rust читает/пишет в SQLite — результат возвращается в React.

```
Пользователь нажимает кнопку
  → React-компонент ловит клик
    → invoke("команда", { id, ... })        ← IPC вызов в Rust
      → Rust Tauri-команда обрабатывает
        → SQLite (autlas.db)            ← единая база данных
          → Результат обратно в React
```

## Хранение данных

Все данные приложения хранятся в **одном файле** SQLite:

```
%AppData%\heavym\autlas\config\autlas.db
```

Раньше данные были разбросаны по нескольким файлам (INI, JSON, TXT). Теперь всё в одном месте — проще бекапить, переносить, и невозможно потерять часть данных.

**Настройки SQLite:**
- **WAL mode** — позволяет читать базу пока идёт запись (не блокирует UI)
- **Foreign keys ON** — удаление скрипта автоматически удаляет его теги
- **busy_timeout 5000ms** — если база занята, ждёт 5 секунд вместо мгновенной ошибки

### Таблицы базы данных

```sql
-- Главная таблица: каждый .ahk файл, который приложение когда-либо видело
scripts
  id            -- UUID v4. Уникальный ID скрипта. НЕ меняется при перемещении файла
  path          -- текущий путь к файлу (lowercase). Меняется при перемещении
  filename      -- имя файла (нужно для поиска "потеряшек" по имени)
  content_hash  -- SHA-256 хеш содержимого файла. Нужен для автоматического
                -- связывания перемещённых файлов (если хеш совпал — это тот же скрипт)
  first_seen    -- когда скрипт впервые обнаружен
  last_seen     -- когда последний раз видели при сканировании
  last_run      -- когда последний раз запускали через приложение или трей
  is_orphaned   -- 1 = файл пропал с диска (удалён или перемещён)
  orphaned_at   -- когда пропал
  -- Поля для будущего маркетплейса (пока пустые):
  community_id, author, version, source_url

-- Теги скриптов (связь многие-ко-многим)
-- Если удалить скрипт из таблицы scripts — его теги удалятся автоматически (CASCADE)
script_tags
  script_id     -- ссылка на scripts.id
  tag           -- название тега (например "PoE", "утилиты")

-- Метаданные тегов: иконка и порядок сортировки в сайдбаре
tag_meta
  tag           -- название тега
  icon          -- имя иконки (например "alarm", "si:discord")
  sort_order    -- порядок в списке (пользователь может перетаскивать)
  color         -- будущее: цвет тега

-- Кеш SVG-иконок, скачанных через Iconify API
-- Чтобы не качать повторно при каждом запуске
icon_svg_cache
  name          -- ключ иконки (например "alarm" или "si:discord")
  bold          -- SVG-путь для bold-варианта
  fill          -- SVG-путь для fill-варианта

-- Папки, которые пользователь пометил как скрытые
hidden_folders
  path          -- путь к папке

-- Корневые папки для сканирования .ahk файлов
scan_paths
  path          -- путь к папке

-- Настройки приложения (ключ-значение)
-- Примеры: close_to_tray=true, migration_complete=1
settings
  key, value
```

### Индексы

Индексы ускоряют поиск по часто используемым полям:

- По **хешу** — для быстрого поиска orphan по содержимому файла
- По **имени файла** — для поиска orphan по имени
- По **статусу orphan** — для фильтрации активных/потерянных скриптов
- По **тегу** — для быстрого поиска "все скрипты с тегом X"

## Идентификация скриптов

### Проблема (было)

Раньше скрипт идентифицировался по **пути к файлу**. Переместил файл — потерял все теги. Переименовал — потерял. Это было главной болью.

### Решение (стало)

Каждый скрипт получает **UUID v4** при первом обнаружении. Этот ID:
- Никогда не меняется
- Хранится в базе данных, а НЕ внутри файла скрипта
- Не зависит от пути, имени или содержимого файла

**Путь** — это просто "адрес" файла. Адрес может меняться, но "паспорт" (UUID) остаётся.

### Что происходит при изменениях

| Что изменилось | Что делает приложение | Теги |
|----------------|----------------------|------|
| Только содержимое (отредактировал код) | Обновляет хеш, UUID тот же | Сохранены |
| Только путь (переместил файл) | Автоматически связывает по хешу | Сохранены |
| Только имя (переименовал) | Автоматически связывает по хешу | Сохранены |
| Путь + имя (переместил + переименовал) | Автоматически связывает по хешу | Сохранены |
| Путь + содержимое (переместил + отредактировал) | Спрашивает пользователя (совпало имя) | Сохранены после подтверждения |
| Имя + содержимое (переименовал + отредактировал) | Считает новым скриптом | Потеряны |
| Всё сразу | Считает новым скриптом | Потеряны |

**Почему "имя + содержимое" = потеря?** Потому что у приложения не остаётся ни одной зацепки: хеш другой (файл изменён), имя другое (файл переименован). Это по сути новый файл. Единственный реалистичный сценарий — одновременно переименовать И переписать файл, что почти никогда не происходит.

## Reconciliation (сверка)

Reconciliation — это процесс сопоставления файлов на диске с записями в базе данных. Запускается **только при ручном Refresh** (кнопка обновления), НЕ при каждом запуске приложения.

### Как работает

```
Фаза 0: Убрать дубли
  - Scan paths могут пересекаться (C:\Scripts и C:\Scripts\Utils)
  - Все найденные пути собираются в HashSet — дубли отсекаются
  - Symlinks и junction points резолвятся через canonicalize()

Фаза 1: Пометить "потеряшек" (orphans)
  - Для каждого скрипта в базе: существует ли файл на диске?
  - Если нет — помечаем is_orphaned = 1

Фаза 2: Найти совпадения для новых файлов
  Для каждого .ahk на диске, которого нет в базе:
    1. Совпадает хеш? → Автоматически связать (молча, без вопросов)
       Это значит файл был перемещён без изменения содержимого
    2. Совпадает имя файла? → Показать уведомление, спросить пользователя
       Это значит файл был перемещён И отредактирован
    3. Ничего не совпало? → Создать новый UUID
       Это реально новый скрипт

Автоматическое удаление orphans ОТКЛЮЧЕНО.
  Скрипты на отключённых USB-дисках или сетевых папках могут выглядеть
  как "потерянные", но на самом деле они просто временно недоступны.
  Пользователь сам решает — связать или удалить через UI.
```

### Оптимизация производительности

Reconciliation должна быть быстрой даже на 1000+ скриптов:

- **Уже известные скрипты** — только обновляем `last_seen` (один SQL UPDATE на всех)
- **Хеш пересчитывается** только если файл реально изменился (сравниваем mtime файла с last_seen в базе)
- **Все данные загружаются одним SQL-запросом** (не по одному на скрипт)
- **Mutex отпускается** на время чтения файлов с диска (чтобы не блокировать другие команды)

Результат: ~50ms вместо 5 секунд на 1000 скриптов.

## Rust-бекенд

### Модули

```
src-tauri/src/
  lib.rs        — Tauri-команды, трей, watcher. Точка входа.
                  Команды — тонкие обёртки вокруг db.rs и reconcile.rs
  db.rs         — Вся работа с SQLite: создание таблиц, чтение, запись
  reconcile.rs  — Алгоритм сверки файлов с базой данных
  migrate.rs    — Одноразовая миграция из старого INI-формата в SQLite
```

### Состояние приложения

Rust хранит два объекта в памяти, доступных всем командам:

```rust
DbState(Mutex<Connection>)              — подключение к SQLite
TraySettingsState(Mutex<TraySettings>)   — настройка "сворачивать в трей"
```

`Mutex` гарантирует что только один поток одновременно пишет в базу.

### Основные команды

| Команда | Параметры | Что делает |
|---------|-----------|------------|
| `get_scripts` | `force_scan` | false = быстрая загрузка из БД, true = полный скан диска + reconciliation |
| `save_script_tags` | `id, tags` | Заменить все теги скрипта (атомарная транзакция) |
| `add_script_tag` | `id, tag` | Добавить один тег |
| `remove_script_tag` | `id, tag` | Удалить один тег |
| `rename_tag` | `old, new` | Переименовать тег у всех скриптов (транзакция) |
| `delete_tag` | `tag` | Удалить тег у всех скриптов + метаданные (транзакция) |
| `resolve_orphan` | `orphan_id, action` | Связать потерянный скрипт с новым путём или удалить |
| `get_script_meta` | `path` | Вернуть хеш, дату создания, изменения, последнего запуска |
| `run_script` | `path` | Запустить скрипт + записать last_run |
| `restart_script` | `path` | Убить + запустить заново + записать last_run |

### Фоновый мониторинг процессов (Watcher)

Отдельный поток, проверяет запущенные процессы каждые 1.5 секунды:

- **Переиспользует** объект `System` (не создаёт новый каждый раз — экономит память)
- Вызывает только `refresh_processes()` (не трогает CPU, память, диски)
- Если скрипт запустился/остановился — отправляет событие `script-status-changed` на фронтенд
- Обновляет меню в трее и нативный Win32-попап

### Трей-попап

Нативный Win32 попап на GDI+ (не webview — мгновенный, легковесный).
Показывает запущенные скрипты с кнопками Stop / Restart / UI.
Restart из трея тоже записывает `last_run` в базу.

## Фронтенд

### Поток данных

```
Запуск приложения (быстрый, ~400ms):
  → useScriptTree.fetchData(false)
    → invoke("get_scripts", { forceScan: false })
      → Rust: читает список путей и теги из SQLite (без скана диска)
    → Скрипты сразу отображаются на экране

Ручное обновление (по кнопке Refresh):
  → useScriptTree.fetchData(true)
    → invoke("get_scripts", { forceScan: true })
      → Rust: сканирует диск → reconciliation → обновляет базу
    → Если найдены перемещённые скрипты:
      → событие orphan-matches-found
        → появляется toast "Обнаружены перемещённые скрипты"
          → пользователь нажимает "Обзор" → модалка с выбором Link/Skip
```

### Ключевые файлы

```
src/
  App.tsx                     — корневой компонент: тосты, drag-drop, модалки
  api.ts                      — интерфейс Script (id, path, tags, ...)
  hooks/useScriptTree.ts      — загрузка данных, операции с тегами, Vim-навигация
  store/useTreeStore.ts       — zustand-стор: состояние UI (pending, editing, focus)
  components/
    OrphanReconcileDialog.tsx  — модалка "Обнаружены перемещённые скрипты"
    ScriptDetailPanel.tsx      — панель деталей: мета-инфо (ID, хеш, даты)
    TagIconPicker.tsx          — выбор иконки (336 встроенных + Iconify API)
    SettingsPanel.tsx          — настройки (авто-рефреш, Everything, пути)
```

### События (Rust → Фронтенд)

Rust отправляет события через Tauri IPC, фронтенд слушает через `listen()`:

| Событие | Данные | Когда срабатывает |
|---------|--------|-------------------|
| `script-status-changed` | `{path, is_running, has_ui}` | Watcher заметил запуск/остановку скрипта |
| `script-tags-changed` | `{id, tags}` | Теги скрипта изменились (из любого источника) |
| `orphan-matches-found` | `PendingMatch[]` | Reconciliation нашла перемещённые файлы |
| `scan-progress` | `number` | Прогресс сканирования диска (WalkDir) |

### Операции с тегами

Все операции с тегами используют **UUID скрипта** (`id`), НЕ путь к файлу:

```typescript
// Так правильно:
invoke("add_script_tag", { id: script.id, tag })

// Так было раньше (больше не работает):
invoke("add_script_tag", { path: script.path, tag })  // УСТАРЕЛО
```

Это ключевое изменение — теги привязаны к ID, а не к пути. Поэтому перемещение файла не ломает теги.

### Система уведомлений (Toasts)

Библиотека **Sonner**. Плавные анимации входа/выхода, автоматический стекинг.

Три типа тостов с фиксированными ID (не дублируются при повторных вызовах):
- `"scan"` — статус сканирования ("Сканирование..." → "Библиотека синхронизирована")
- `"orphan"` — "Обнаружены перемещённые скрипты" с кнопкой "Обзор"
- `"everything"` — статус Everything Search ("Запущен" / "Не установлен")

## Миграция со старого формата

При первом запуске новой версии автоматически мигрирует данные из INI в SQLite.

Происходит один раз, внутри транзакции (если что-то пойдёт не так — откатится):

1. Прочитать `manager_data.ini` (кодировка UTF-16 LE)
2. Перенести: скрипты+теги, иконки тегов, порядок тегов, скрытые папки, пути сканирования, настройки
3. Вычислить SHA-256 хеш для каждого существующего файла
4. Перенести `icon_cache.json` в таблицу `icon_svg_cache`
5. Поставить флаг `migration_complete = "1"` (чтобы не мигрировать повторно)
6. Сохранить старый INI как `.bak` (на всякий случай)

## Безопасность и надёжность

### Транзакции

Все операции, состоящие из нескольких SQL-запросов, обёрнуты в транзакции:
- **Замена тегов** — DELETE + INSERT атомарны. Если крашнется между ними — откатится
- **Переименование тега** — обновление в script_tags + tag_meta за одну транзакцию
- **Удаление тега** — удаление из script_tags + tag_meta за одну транзакцию

### Обработка конфликтов

`upsert_script` обрабатывает два типа конфликтов:
- **Совпал UUID** — обновить запись (скрипт переместился)
- **Совпал путь** — обновить запись (два процесса увидели один файл)

### Конкурентный доступ

- **busy_timeout = 5 секунд** — если база занята записью, повторяет попытку
- **Mutex отпускается** во время чтения файлов с диска (чтобы теги можно было добавлять пока идёт скан)
- **Poisoned mutex** — если поток упал с ошибкой, mutex восстанавливается через `into_inner()` вместо краша

### Обработка ошибок

Reconciliation пробрасывает все ошибки наверх через `?` (оператор Result).
Никаких `let _ =` — если SQLite вернул ошибку, пользователь увидит её.
