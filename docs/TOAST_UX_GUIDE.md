# Toast UX Guide — практический шпаргалочник

Сфокусированный справочник для единой системы тостов AHK Manager
(success / warning / error / info).
Источники в скобках: **MD** = Material Design 2/3, **HIG** = Apple Human Interface
Guidelines, **Fluent** = Microsoft Fluent / WinUI, **NNG** = Nielsen Norman Group,
**Atlassian** = Atlassian Design System (Flag), **Primer** = GitHub Primer,
**Sonner** = дефолты библиотеки Sonner, **WAI-ARIA** = W3C ARIA Authoring Practices,
**Smashing** = Smashing Magazine.

---

## 1. Выбери правильную поверхность

Правило из **NNG** («Indicators, Validations, and Notifications»):

| Тип фидбэка                                          | Поверхность      | Почему                                                                |
| ---------------------------------------------------- | ---------------- | --------------------------------------------------------------------- |
| Пассивное подтверждение системного события          | **Тост**         | Исчезает, не блокирует поток (NNG)                                    |
| Валидация привязанная к конкретному инпуту          | **Inline**       | Должно быть рядом с полем (NNG)                                       |
| Срочное, требует действия, блокирует задачу         | **Modal-диалог** | Должно прервать и потребовать решения (NNG, WAI-ARIA `alertdialog`)   |
| Постоянный контекст уровня приложения (offline, апд)| **Баннер**       | Висит пока условие не разрешится (Atlassian Section message, Primer)  |
| Обучение / онбординг по фиче                        | **Coach mark**   | Полу-постоянный, привязан к таргету (Fluent TeachingTip)              |

**Жёсткое правило (NNG):** «Тост … был бы плохим способом показать ошибку», если
от пользователя требуется действие. Если пользователю *нужно* что-то сделать —
это не тост.

**Жёсткое правило (Fluent TeachingTip docs):** временные поверхности «не рекомендуются
для оповещения об ошибках или важных изменениях статуса».

Для нашего приложения:

- Скан завершён, скрипт запущен/убит, тег добавлен → **тост**.
- «Папки не существует» при вводе в настройках → **inline** под полем.
- «Несохранённые изменения, всё равно выйти?» → **модалка**.
- «AutoHotkey не установлен / неправильная версия» → **постоянный баннер** сверху окна.

## 2. Длительность

Авторитетные цифры:

- **MD2 snackbar spec:** 4с по умолчанию, диапазон 4–10с, максимум 10с.
  Никогда меньше 4с (пользователь не успеет прочитать).
- **Sonner default:** 4000 мс. Errors и warnings на практике часто поднимают до 5–6с.
- **Atlassian Flag:** info/success автоисчезают через ~8с; **warning и error по умолчанию
  persistent** (требуют явного закрытия).
- **WCAG 2.2.3** (цит. в WAI-ARIA Alert pattern): авто-исчезновение должно давать
  «достаточно времени для ответа пользователя»; если информация критична — не
  исчезать автоматически.

Рекомендация для AHK Manager:

| Тип         | Длительность                  | Persistent?                       |
| ----------- | ----------------------------- | --------------------------------- |
| **success** | 3–4 с                         | Нет                               |
| **info**    | 4–5 с                         | Нет                               |
| **warning** | 6–8 с                         | Опционально — persist если actionable |
| **error**   | **Persistent** (без таймера)  | **Да** — явный dismiss            |

Pause-on-hover и pause-on-focus ожидаются (Sonner, MD). Перезапускать таймер
когда курсор уезжает.

## 3. Позиция

| Позиция            | Плюсы                                                  | Минусы                                          | Кто использует           |
| ------------------ | ------------------------------------------------------ | ----------------------------------------------- | ------------------------ |
| **Bottom-center**  | Не перекрывает контент; близко к большому пальцу на mobile | Может конфликтовать с FAB / status bar       | MD snackbar (default)    |
| **Bottom-right**   | Не мешает на десктопе, рядом с system tray             | Можно пропустить если смотришь наверх           | Fluent, Sonner default, Primer, большинство десктопов |
| **Top-right**      | Зеркалит OS notification center (Win 10/11, macOS)     | Конкурирует с title bar / window controls       | VS Code, многие IDE      |
| **Top-center**     | Высокая видимость                                      | Навязчиво, перекрывает header / breadcrumbs (NNG предупреждает) | Некоторые веб-апп |

**Рекомендация для AHK Manager (десктопный Tauri):** **bottom-right**, стек растёт
вверх. Совпадает с Windows-конвенциями, не мешает дереву скриптов и тулбару, и
совпадает с дефолтом Sonner который мы скорее всего уже используем.

Никогда не давай тостам перекрывать главные действия или поля ввода в которых
пользователь сейчас печатает (MD).

## 4. Стек и переполнение

- **MD2 явное правило:** «Только один snackbar может отображаться одновременно».
  Новые заменяют старые или встают в очередь.
- **Sonner default:** `visibleToasts = 3`. Старые тосты сворачиваются в стек и
  раскрываются по hover.
- **Atlassian Flag:** стек вертикально, новейший наверху, старые опускаются;
  ограничивает видимое количество и использует «show more».

Рекомендация:

- Максимум **3 видимых** одновременно.
- Новый тост появляется внизу стека (ближе к углу) и толкает старые наверх,
  ИЛИ новейший наверху — выбери один и держись. Sonner использует
  newest-on-top со сворачиванием.
- При переполнении **сворачивать** старые в стопку, раскрываемую по hover.
- **Дедупликация**: если тот же тост (тот же id/key) сработал дважды — увеличить
  счётчик («Saved x3») вместо складирования дублей (паттерн Sonner, также
  рекомендация Smashing).
- **Coalesce bursts**: если 20 status-событий выстрелили за 500мс — показать
  один summary-тост.

## 5. Закрываемость

Правила синтезированные из **MD**, **Atlassian**, **WAI-ARIA**, **NNG**:

- **success / info**: автоисчезновение ок; всё равно дай кнопку Close для тех
  кто хочет действовать быстро или использует screen reader.
- **warning**: автоисчезновение только если пользователь может его безопасно
  игнорировать; иначе persistent с кнопкой Close.
- **error**: **всегда требовать явного dismiss.** Auto-dismiss errors — это
  WCAG 2.2.3 fail (WAI-ARIA Alert pattern) и top-антипаттерн (Smashing, NNG).
- Каждый тост должен закрываться с клавиатуры (Esc или фокусируемая кнопка
  Close). Sonner экспонирует глобальный хоткей (по умолчанию `Alt+T`) который
  фокусирует область тостов — стоит зеркалить.

## 6. Action-кнопки

**Правило MD2:** snackbar может содержать **максимум одно действие**, и оно не
должно быть единственным способом выполнить это действие (у пользователя должен
быть другой путь в UI).

Хорошие действия:

- **Undo** — золотой стандарт. Лучшее для деструктивных операций (delete,
  archive, remove tag). Сочетай с optimistic UI: выполняй действие сразу,
  показывай тост с Undo на ~5–10с, коммить только при dismiss. (Gmail, Sonner
  promise toasts.)
- **Retry** — на временный сбой (network, file lock). Парь с error-тостом.
- **Open / Review / View** — на завершение фоновой задачи («Скан завершён — View»).
- **Copy details** — на ошибки, чтобы юзер мог вставить в баг-репорт.

Плохие действия (антипаттерны):

- Больше одной кнопки (MD).
- Деструктивные действия в transient-тосте — он исчезает раньше чем юзер успевает подумать.
- Действие — *единственный* способ добраться до фичи (MD).
- Длинные лейблы кнопок — держи 1–2 слова (MD: ALL CAPS в MD2, sentence-case в MD3).

## 7. Доступность (a11y)

Из **WAI-ARIA Alert pattern** и реализаций Sonner / Primer:

| Тип тоста             | ARIA role                            | aria-live    | Заметки                                       |
| --------------------- | ------------------------------------ | ------------ | --------------------------------------------- |
| success / info        | `role="status"`                      | `polite`     | Не прерывает текущую речь                     |
| warning               | `role="status"` или `role="alert"`   | `polite`     | Polite кроме safety-critical                  |
| error                 | `role="alert"`                       | `assertive`  | Прерывает; в паре с persistent dismiss        |

Критические правила:

- **Никогда не переводи фокус** на тост автоматически — WAI-ARIA Alert pattern
  явно запрещает: «крайне важно чтобы они не влияли на keyboard focus».
- Контейнер тостов должен существовать в DOM с самого старта приложения (live
  region). Не добавляй live region одновременно с сообщением — screen readers
  могут пропустить.
- Дай способ **добраться** до тостов с клавиатуры (Sonner использует `Alt+T`
  чтобы сфокусировать регион).
- Кнопке Close нужно accessible name (`aria-label="Dismiss notification"`).
- Уважай `prefers-reduced-motion`: пропускай slide/scale-анимацию, только opacity.
- Не auto-dismiss errors (WCAG 2.2.3).
- Цвет — не единственный сигнал: каждому типу нужна своя иконка (MD, Primer,
  Atlassian — все требуют).

## 8. Антипаттерны

Скомпилировано из **Smashing**, **NNG**, **MD**, **Atlassian**:

- **Тостить ошибки которые автоисчезают.** Юзер моргнул, пропустил, приложение
  ощущается как «с привидениями». Ошибки должны persist.
- **Тостить ошибки валидации формы.** Валидация принадлежит **inline** рядом с
  полем (NNG). Тост «В форме ошибки» без подсветки поля — враждебно.
- **Тосты как единственный фидбэк критического события.** Если потерять
  сообщение значит потерять данные — это не должно быть тостом.
- **Стек из 8 тостов друг на друге.** Лимит 3, остальное сворачивать.
- **Тост перекрывает кнопку которую юзер только что нажал** или поле в которое печатает.
- **Длинная проза.** MD: одна-две короткие строки; Fluent: пользователи читают
  3–5 слов в среднем.
- **Несколько action-кнопок.**
- **Auto-dismiss < 4с.** Ниже читаемого порога (MD).
- **Тосты которые блокируют клики** под собой. Тосты должны быть non-modal —
  клики проходят сквозь в приложение (Smashing, MD).
- **Тосты top-center на десктопе** — конфликтуют с title-барами и ощущаются как
  browser permission prompts.
- **Агрессивная анимация in/out** — раздражает и a11y-проблема (Smashing, WCAG 2.3.3).
- **Идентичные дубликаты.** Coalesce или counter-stack.
- **Угон фокуса** в тост (WAI-ARIA).

## 9. Не тостить про…

- **Валидацию форм.** Inline, рядом с полем. (NNG)
- **Деструктивные подтверждения** («Удалить 200 файлов?»). Modal/confirm dialog. (NNG)
- **Постоянное состояние системы** («Offline», «AHK не найден», «Доступно
  обновление»). Используй баннер который висит пока условие не разрешится.
  (Atlassian, Primer)
- **Онбординг / discovery фичи.** Coach mark или teaching tip привязанный к
  элементу. (Fluent TeachingTip)
- **Критические ошибки которые блокируют задачу пользователя.** Modal alert
  dialog. (WAI-ARIA `alertdialog`)
- **Долгие операции с прогрессом.** Прогресс-бар, status area, или
  *progress-тост* который обновляется на месте (Sonner `toast.promise`) — не
  поток отдельных тостов.
- **Маркетинг / upsell.** Просто не надо. (Smashing)

## 10. Накопительный / батчевый фидбэк

Вопрос «5 файлов скопировано».

Паттерны из **Sonner**, **Atlassian**, и конвенций Gmail/VS Code:

- **Один батчевый тост на одно действие пользователя.** Если юзер кликнул «Run
  scan» один раз — выстрели **один** тост в конце: *«Скан завершён — 134
  скрипта за 1.2с»* (мы уже так делаем — оставляем).
- **Promise / progress тост.** Для операций с явным началом и концом — показывай
  один тост который проходит три состояния на месте: `loading → success | error`.
  Sonner отгружает `toast.promise` именно для этого. Рекомендуется для: скан,
  bulk tag rename, bulk enable/disable, restart-all.
- **Counter coalescing.** Если фоновые события стреляют повторно (например
  «Script X exited» дважды за 300мс) — слить в «Script X exited (×2)» вместо
  двух отдельных тостов.
- **Окно агрегации.** Для burst-событий — debounce ~500–800 мс и эмить один
  summary. Пример: hot-reload watcher детектит 12 file changes — один тост:
  «12 файлов изменено, пересканировано».
- **Per-item тосты только если каждый item значим независимо** и от юзера
  ожидается действие на каждый. Иначе батчить.
- **Никогда** не генерируй один тост на файл при копировании 100 файлов. Это
  notification flood — классический антипаттерн (Smashing).

## Quick reference card для AHK Manager

```
SUCCESS  3–4с  bottom-right  role=status  polite     auto-dismiss  опц. close-кнопка
INFO     4–5с  bottom-right  role=status  polite     auto-dismiss  опц. close-кнопка
WARNING  6–8с  bottom-right  role=status  polite     auto-dismiss* close-кнопка обязательна
ERROR    ∞     bottom-right  role=alert   assertive  PERSISTENT    close-кнопка обязательна
                                                     + опц. Retry / Copy details

Стек: max 3 видимых, новейший внизу стека (привязка к углу), сворачивать переполнение.
Pause on hover/focus. Esc закрывает сфокусированный тост. Alt+T фокусирует регион тостов.
Дедуп идентичных сообщений с ×N счётчиком. Дебаунсить burst (~500мс) в один summary.
Никогда не двигать фокус. Никогда не блокировать клики. Никогда не тостить валидацию форм.
```

\* warnings auto-dismiss только если non-actionable; persist если юзер должен ответить.
